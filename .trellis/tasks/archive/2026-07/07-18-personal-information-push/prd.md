# 优化个人信息推送产品

## Goal

把当前“面向创作者的 AI 实践雷达”优化为真正供个人每天使用的信息推送产品：用户配置关注主题，系统从稳定的国内外来源收集信息，优先展示可信、较新、相关且值得行动的内容，并在固定时间发送不重复的精简摘要。AI 是默认主题，但不是硬编码边界；内容创作保留为可选动作，不再定义产品主路径。

## Background

- 当前链路已具备 Python 采集、Go 入库与分析、React 看板、飞书推送及内容生成。
- 新雷达只启用 DEV Community、GitHub、Bluesky、Reddit，且采集查询与资格判断写死为 AI/Agent/Skill/MCP；旧关键词管理尚未接入新雷达。
- 当前看板围绕“待实践 → 已实践 → 三平台创作”，不适合作为通用的个人信息收件箱。
- 当前摘要按数据库返回顺序取前八条，跨来源热度不可比较，没有主题多样性约束，也会在早晚报重复发送同一信号。
- 已确认的用户偏好：AI 为默认但允许其他主题；中文为主、中英信息均可；国内外来源一视同仁；重视原始证据和落地价值；每日 08:00、18:00 摘要，真正重磅消息即时提醒。
- 2026-07-18 基线：Go 全量测试、Python 8 项测试、前端 6 项测试、前端构建和 lint 均通过。

## Requirements

### R1. 可配置关注主题

- 复用现有关键词数据和 CRUD，把它作为新雷达的“关注主题”配置入口。
- 首次安装默认创建 AI 主题；用户可新增、停用和删除其他主题，无需修改代码。
- 第一版最多启用 10 个主题，避免搜索 API 请求数和噪声无上限增长。
- 主题调整只影响后续采集和分析，不自动重跑历史信号。

### R2. 稳定且可控的信息来源

- 保留 DEV、GitHub、Bluesky、Reddit 及现有单源失败隔离。
- 新增 RSS/Atom 来源，允许管理员配置最多 20 个 HTTP(S) Feed；使用 Feed 自带内容作为证据，不实现任意网页正文爬虫。
- GitHub 使用官方 REST API：按主题发现近期活跃仓库，并允许配置明确的关注仓库列表以监控新 Release。
- GitHub 公开数据允许无 Token 使用；配置 Token 时使用更高额度。采集必须串行、有限量，并正确暴露限流失败，不能无限重试。
- 不因来源国家、语言或平台类型直接加减优先级。

### R3. 证据优先的筛选与分析

- 资格判断基于启用主题，而不是固定 AI 常量；默认 AI 主题保留常见中英文别名匹配。
- 明确关注仓库的新 Release 即使标题未重复主题词，也可以进入分析。
- 每条入选信息必须保留原始链接、证据类别和证据快照。
- AI 分析新增 `matchedTopics` 和 1–5 的 `valueScore`；分数综合个人相关性、信息新意和行动价值，不能替代事实验证。
- 无法匹配任何启用主题且不是明确订阅更新的内容，在调用模型前拒绝。

### R4. 可解释的排序与去重

- 看板和摘要共享同一套确定性排序：AI 价值分、时效加分、重磅加分；原始平台热度不得直接跨来源比较。
- 排序结果向前端返回 `rankScore` 和匹配主题，用户可以理解内容为何出现。
- 精确相同链接或规范化标题相同的内容在同一结果集中只出现一次。
- 摘要最多 8 条，每个主题最多 2 条；质量不足时允许少于 8 条或不发送。

### R5. 个人信息收件箱

- 主界面改为“收件箱 / 稍后处理 / 已处理”，并保留“不感兴趣”。
- 迁移现有状态：`new → inbox`、`queued → saved`、`practiced → done`、`dismissed` 保持不变。
- 被忽略的内容不再占用看板列表上限或进入推送。
- 第一版不做自动兴趣学习；用户反馈只立即改变该条信息的状态，避免伪个性化。

### R6. 不重复的摘要与提醒

- 保留北京时间每三小时采集、每日 08:00/18:00 摘要和每天最多三条重磅提醒。
- 成功推送后记录信号的最后推送时间；同一信号不得再次进入摘要或重磅提醒。
- 通知失败时不得标记为已推送，后续可以重试。
- 摘要只发送尚未处理、未忽略、未推送的高价值信号。

### R7. 内容创作降级为可选能力

- 已处理信号可按需生成现有小红书、公众号和 X 内容包，但首页不再把创作作为主要目标。
- 继续保留来源链接和第一人称实测防伪规则；没有用户验证证据时不得生成“亲测”等表述。
- 本任务不增加新的发布平台或自动发帖能力。

### R8. 工程质量与兼容性

- 复用现有关键词、来源配置、证据、分析、调度、通知和内容生成链路，不重写技术栈。
- 新增 RSS 使用 Python 标准库，不增加第三方解析依赖。
- 来源设置、生命周期、分析字段和 API 响应必须在后端、前端类型与测试中同步。
- 删除前端当前入口完全不可达的旧热点/图谱页面及其专用依赖；后端旧表和旧接口本任务暂不删除，以保护已有数据和兼容性。
- 非平凡的主题筛选、GitHub Release 发现、RSS 解析、排序、去重、摘要多样性和推送去重必须有最小测试。

## Acceptance Criteria

- [x] AC1：首次安装存在默认 AI 主题；用户能在主界面新增、停用和删除主题，下一轮采集使用当前启用主题。
- [x] AC2：采集命令不再包含固定的 `mcp/claudecode/agents/ai` 查询；无启用主题时安全跳过主题搜索。
- [x] AC3：管理员可配置 RSS/Atom Feed，合格条目通过现有证据与入库链路展示，单个坏 Feed 不阻塞其他来源。
- [x] AC4：GitHub 按主题发现仓库，并能从明确关注仓库的新 Release URL 创建独立信号；重复 Release 不重复入库。
- [x] AC5：非 AI 主题内容可以被筛选、分析和展示；无主题匹配的普通内容在模型调用前拒绝。
- [x] AC6：看板按收件箱状态分区，支持稍后处理、完成和不感兴趣；刷新后状态保持。
- [x] AC7：看板和摘要使用同一排序函数；相同链接/标题去重，摘要满足总数与单主题上限。
- [x] AC8：成功发送后的信号不会在后续早报、晚报或重磅提醒中重复发送；发送失败仍可重试。
- [x] AC9：内容生成仍可从已处理信号进入，证据安全规则保持不变，但首页主文案和布局不再围绕小红书或三平台创作。
- [x] AC10：Go、Python、前端测试、前端构建和 lint 全部通过；关键主路径完成浏览器验收。
- [x] AC11：README 与产品文案反映“个人信息雷达”及当前真实来源，不再宣称只服务海外 AI 项目或固定创作平台。

## Out of Scope

- 任意网页通用爬虫、绕过登录/反爬或没有稳定接口的网站逐站适配。
- GitHub Discussions GraphQL、全量公共 Events、自动导入全部 Stars；明确关注仓库已经覆盖第一版更新需求。
- 多租户、团队协作、公开 SaaS、付费订阅、移动 App 或浏览器插件。
- 基于行为训练推荐模型或自动修改兴趣权重。
- 删除后端旧热点、图谱、旧调度表和历史数据；待新链路稳定后另行清理。
- 新增内容平台、自动发布或图片生成。
